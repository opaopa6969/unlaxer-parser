# Part A: tinyexpression 専用 IDE — 独立プロジェクトとしての設計

## テーマ
tinyexpression の式を書く・検証する・管理するための専用 IDE を独立プロジェクトとして設計する。ターゲットユーザー、デプロイ形態、アーキテクチャ、MVP の範囲を議論する。

---

## Scene 1: そもそも専用 IDE って誰が使うのか

今泉: 「そもそも専用 IDE って誰が使うんですか？ VSCode の拡張があるのに、わざわざ別のものを作る理由は？」

ヤン: 「tinyexpression の式を書くのはプログラマだけじゃない。保険会社の料率担当者、銀行のリスク計算担当者 — 彼らは VSCode を起動したことがない人たちだよ。match式で 470 ケースの料率テーブルを管理してる人に『まず VSCode をインストールして、拡張機能を入れて、ワークスペース設定をして...』なんて言えない。」

今泉: 「要するに、非プログラマがビジネスルールを編集する道具ってことですか？」

ヤン: 「そう。Excel の代わりに tinyexpression の式を使ってる現場がある。今は式をテキストファイルで管理してる。それを安全に編集・テストできる環境が欲しい。」

千石: 「VSCode は重い、設定が難しい、というだけではありません。問題の本質は、VSCode が汎用エディタであることです。式のプレビュー、変数の試行、依存グラフの表示 — これらは VSCode の拡張では限界があります。専用の UI が必要です。」

  → **Gap 発見: ターゲットユーザーの具体的なペルソナが未定義。「非プログラマ」は広すぎる。料率担当者？ 業務ルール管理者？ データアナリスト？**

今泉: 「他にないの？ Web-based にするか、Electron にするか、IntelliJ プラグインにするか。選択肢は？」

リヴァイ: 「IntelliJ プラグインは論外だ。非プログラマに IntelliJ を使わせる気か。選択肢は Web か Electron か Tauri だ。」

ヤン: 「Web-based が最もデプロイが楽だね。URL を開くだけで使える。でもオフラインで動く必要があるなら Electron か Tauri。」

  → **Gap 発見: オフライン要件が未定義。ユーザーの作業環境にインターネット接続がない場合があるか？**
  → **Gap 発見: セキュリティ要件が未定義。ビジネスルール（料率テーブル等）は機密情報の可能性がある。クラウド経由は許容されるか？**

千石: 「デプロイ形態を決める前に、ターゲット企業の IT ポリシーを理解すべきです。金融機関なら Web アプリのインストールすら制限されていることがある。」

  → **Gap 発見: デプロイ形態がターゲットユーザーの IT 環境に依存する。調査が必要。**

---

## Scene 2: IDE に必要な機能は？

千石: 「IDE に必要な機能を整理しましょう。まず最低限必要なものは何ですか？」

ヤン: 「式のリアルタイムプレビューだね。左に式を書いて、右に即座に結果が出る。これがないと IDE を作る意味がない。」

千石: 「それだけでは足りません。変数の値を入力して試せるサンドボックスが必要です。`$age` や `$income` の値を入力フォームで設定して、結果を確認する。Excel のセル参照のような直感的な体験です。」

リヴァイ: 「match 式の 470 ケースをどうする。全部表示したら画面が埋まるぞ。」

千石: 「折りたたみ表示は必須です。さらに、特定の条件で絞り込める検索機能。470 ケースの中から `age >= 65` に該当するケースだけを表示する、といった機能です。」

  → **Gap 発見: match 式の大量ケースの管理 UI が未定義。折りたたみ、検索、ソート、グルーピングの仕様が必要。**

今泉: 「FormulaInfo のメタデータってありますよね。名前とか dependsOn とか。あれはどう編集するんですか？」

ヤン: 「UBNF のアノテーションで書くのは非プログラマには無理だから、フォーム形式のエディタが要るね。名前、説明、依存先をフォームで入力して、裏で正しい構文を生成する。」

  → **Gap 発見: FormulaInfo メタデータのフォームエディタの仕様が未定義。どのフィールドを編集可能にするか？ バリデーションは？**

千石: 「dependsOn の依存グラフ可視化も必要です。式 A が式 B に依存し、式 B が式 C に依存する — この関係がグラフで見えなければ、循環依存やデッドコードを見逃します。」

  → **Gap 発見: 依存グラフ可視化のインタラクション仕様が未定義。ノードクリックで式にジャンプ？ 循環依存のハイライト？**

リヴァイ: 「既存の LSP にある機能と、IDE 固有の機能を分けろ。LSP で対応できるものは LSP を使え。二重実装するな。」

ヤン: 「LSP が提供しているのは、補完、診断、ホバー情報、定義ジャンプ。リアルタイムプレビュー、サンドボックス、依存グラフは LSP の範囲外だよ。」

  → **Gap 発見: LSP で提供する機能と IDE 固有機能の境界が明確でない。機能一覧を作ってどちらが担当するか決めるべき。**

---

## Scene 3: プロジェクト構成はどうする

リヴァイ: 「リポジトリ構成を決めろ。既存の tinyexpression-vsix に押し込むのか、別リポジトリか。」

ヤン: 「別リポジトリ `tinyexpression-ide` だろうね。VSIX は VSCode のエコシステムに依存してる。IDE は全く別のフロントエンドだ。」

千石: 「ただし、LSP サーバーは共有すべきです。既存の VSIX の `server-dist/` にある LSP サーバーを使い回す。ロジックの二重管理は品質の敵です。」

リヴァイ: 「共有するなら LSP サーバーのパッケージングを見直す必要があるぞ。今は VSIX にバンドルされてるだけだ。npm パッケージとして独立させるか、別の配布方法が要る。」

  → **Gap 発見: LSP サーバーの独立パッケージング方法が未定義。npm パッケージ化？ JAR 配布？ Docker イメージ？**

今泉: 「そもそも LSP サーバーは Java で書かれてるんですよね。フロントエンドが Web なら、どうやって通信するんですか？」

ヤン: 「LSP は元々 stdio ベースのプロトコル。Web で使うなら WebSocket ブリッジが要る。Monaco Editor がやってるのと同じパターンだよ。」

  → **Gap 発見: LSP の stdio → WebSocket ブリッジの実装方式が未定義。既存ライブラリ（vscode-ws-jsonrpc）を使うか、自前実装か。**

リヴァイ: 「フロントエンドの選択肢を絞れ。Monaco Editor（Web）か、Electron か、Tauri か。」

千石: 「Monaco Editor は VSCode のエディタコンポーネントそのものです。LSP との統合は実績がある。Web ベースで始めて、必要なら Electron でラップする、という段階的アプローチが妥当です。」

ヤン: 「Tauri は Rust ベースで軽量だけど、Monaco との統合実績が少ない。リスクを取る場面じゃないね。」

  → **Gap 発見: Monaco Editor + Web で始めて Electron ラップする場合の技術的制約が未調査。ファイルシステムアクセス、ローカル LSP プロセスの起動など。**

---

## Scene 4: 最もシンプルに始めるなら

ヤン: 「最もシンプルに始めるなら、Monaco Editor + LSP over WebSocket で HTML 1 枚のプロトタイプだよ。紅茶を飲みながら 1 日で作れる規模に収めたい。」

リヴァイ: 「具体的に何ができれば『動くもの』と言えるんだ。」

ヤン: 「(1) 式を入力できる、(2) LSP の補完と診断が動く、(3) 式を評価して結果が表示される。この 3 つ。」

今泉: 「既存の LSP サーバーを WebSocket 対応にするのは大変じゃないんですか？」

ヤン: 「既存の LSP サーバーは Java で、stdio で動いてる。WebSocket ブリッジは薄いプロキシを挟むだけ。Node.js で 50 行くらい。`vscode-ws-jsonrpc` というライブラリがそのまま使える。」

  → **Gap 発見: LSP サーバーの起動・管理（プロセスライフサイクル）が未定義。Web アプリが開かれたら LSP プロセスを起動？ 常駐？ マルチユーザー時は？**

千石: 「式の評価結果表示は LSP の範囲外です。tinyexpression の Java ライブラリを直接呼ぶ必要がある。Web フロントエンドからどうやって Java の評価エンジンを呼びますか？」

ヤン: 「REST API を 1 つ生やすだけだよ。`POST /eval` で式と変数を送って、結果を返す。LSP サーバーと同じ Java プロセスに相乗りすればいい。」

  → **Gap 発見: 式評価 API の仕様が未定義。入力形式（式テキスト + 変数マップ）、出力形式（値 + 型 + エラー情報）、タイムアウト。**
  → **Gap 発見: 悪意のある式（無限ループ、メモリ爆発）に対するサンドボックス保護が未定義。**

リヴァイ: 「プロトタイプの範囲を明確にしろ。match 式の折りたたみも、依存グラフも、FormulaInfo エディタも、全部後回しだ。」

今泉: 「前もそうだったっけ？ VSCode 拡張を作った時も最初は補完だけから始めたんでしたっけ？」

ヤン: 「その通り。VSIX も最初は diagnostics と completion だけだった。同じ戦略でいい。」

---

## Scene 5: MVP 判断

ヤン: 「MVP は決まったね。Monaco Editor + WebSocket LSP ブリッジ + 式評価 API。HTML 1 枚で動くプロトタイプ。」

千石: 「MVP であっても、式の評価結果が正しいことは保証してください。間違った結果を返すくらいなら、エラーを返すべきです。」

リヴァイ: 「MVP のチェックリスト。(1) Monaco Editor が動く。(2) LSP の補完・診断が動く。(3) 式を評価して結果が表示される。(4) 変数を入力できる。この 4 つだ。」

今泉: 「誰が困るの？ MVP が成功したとして、次に何を追加するかの優先順位は？」

ヤン: 「非プログラマが最初に困るのは match 式の管理だろうね。470 ケースをスクロールするのは地獄だ。MVP の次は match 式の折りたたみ UI。」

千石: 「その次は FormulaInfo のフォームエディタです。メタデータを手書きするのは非プログラマには酷です。」

  → **Gap 発見: MVP 後のロードマップ（Phase 2, 3）の優先順位が仮説段階。ユーザーインタビューで検証すべき。**

# Part B: LLM に unlaxer-parser でパーサー/VSIX を作らせるためのガイド

## テーマ
LLM（Claude, GPT 等）に unlaxer-parser を使って新しい言語のパーサーと VSIX を作らせるための手順書・プロンプトテンプレートを設計する。

---

## Scene 1: LLM に何を渡せばパーサーを作ってくれるのか

今泉: 「そもそも LLM に何を渡せばパーサーを作ってくれるんですか？ README.md を丸ごと渡せばいい？」

ヤン: 「README は概要しか書いてない。LLM が実際にパーサーを書くには、(1) UBNF 文法のサンプル、(2) @mapping, @leftAssoc, @eval の具体的な説明、(3) 生成されるコードの構造 — この 3 つが要る。」

今泉: 「tinyexpression-p4.ubnf がサンプルになるってことですか？」

ヤン: 「そうだけど、p4.ubnf は 470 ケースの match 式がある巨大ファイルだよ。LLM のコンテキストを無駄に消費する。最小限のサンプルが要る。」

  → **Gap 発見: LLM が最初に読むべき「1 ファイル」が存在しない。最小限の UBNF サンプル + アノテーション説明をまとめたファイルが必要。**

千石: 「Quick Start の TinyCalc は良い出発点ですが、@mapping と @eval の説明が不十分です。LLM はアノテーションの意味を正しく理解できないでしょう。」

リヴァイ: 「具体例を見せろ。@mapping がどういう Java コードを生成するか、入力と出力を並べて見せれば LLM は理解する。」

  → **Gap 発見: @mapping → 生成される Java コードの対応表が存在しない。Before/After の例が必要。**
  → **Gap 発見: @eval default が何を自動化するか、何を手動実装にするかの説明が不足。**

今泉: 「要するに、LLM に渡す『チートシート』みたいなものが欲しいってことですか？」

ヤン: 「その通り。1 ファイル、2000 行以内。UBNF 構文、アノテーション、生成コードの対応、よくある間違いと対処法。これが『LLM 向け unlaxer リファレンス』になる。」

  → **Gap 発見: LLM 向けリファレンスドキュメントの構成・フォーマットが未定義。**

---

## Scene 2: LLM が間違えやすいポイント

千石: 「LLM が間違えやすいポイントを整理しましょう。経験から言えることは？」

ヤン: 「一番多いのは Expression の優先順位の間違い。Boolean 式と Number 式の順序を逆にすると、`a > b` が数値比較じゃなくて別のものにパースされる。PEG の ordered choice は順序が全て。」

  → **Gap 発見: Expression の優先順位の正しい並べ方のルール説明が不足。「なぜこの順序か」の理由付きガイドが必要。**

リヴァイ: 「左再帰も間違える。PEG で `Expr = Expr '+' Expr` と書く LLM がいる。これは無限ループだ。」

千石: 「`@leftAssoc` アノテーションの使い方を正確に教えないと、LLM は BNF 風の左再帰を書いてしまいます。」

  → **Gap 発見: 左再帰の回避方法と @leftAssoc の使い方の具体的な説明が不足。NG パターンと OK パターンの対比が必要。**

今泉: 「@mapping の params の順序って重要なんですか？」

ヤン: 「致命的に重要。@mapping の params はパーサーが子ノードを返す順序に対応してる。順序を間違えると、`a + b` で a と b が入れ替わったりする。」

  → **Gap 発見: @mapping params の順序決定ルールが暗黙知。文法のどの部分がどの params に対応するかの明確なルールが必要。**

千石: 「sealed interface の網羅性も問題です。Java 21 の sealed interface を使っている場合、switch 式で全ケースを網羅しないとコンパイルエラーになる。LLM が新しいノード型を追加した時に、既存の switch を更新し忘れる。」

  → **Gap 発見: sealed interface にノード型を追加した時の波及影響チェックリストがない。**

リヴァイ: 「エラーメッセージの読み方も教えるべきだ。unlaxer の MapperGenerator がエラーを出した時、LLM がエラーメッセージを正しく解釈できるか？」

  → **Gap 発見: MapperGenerator のエラーメッセージ一覧と対処法のドキュメントがない。**

---

## Scene 3: 実際の手順を示せ

リヴァイ: 「理屈はいい。手順を示せ。Step 1 から Step 6 まで。」

ヤン: 「こうだね。」

```
Step 1: 言語の具体例を書く（入力と期待出力）
  例: "1 + 2 * 3" → 7
  例: "if(x > 0, x, -x)" → |x|

Step 2: UBNF 文法を書く
  p4-tinycalc.ubnf を参考に
  @mapping, @leftAssoc を追加

Step 3: mvn compile で生成
  Parser, AST, Mapper が自動生成される

Step 4: evalXxx() を実装
  @eval default で大部分は自動生成
  カスタム関数のみ手動実装

Step 5: テスト
  生成された XxxTest.java にテストケース追加

Step 6: VSIX パッケージング
  textmate grammar + LSP サーバー設定
```

今泉: 「Step 2 から Step 3 で失敗した時はどうするんですか？ エラーが出て先に進めない場合。」

リヴァイ: 「そこが一番 LLM がハマるところだ。`mvn compile` のエラーメッセージを読んで文法を修正する。だがエラーメッセージが分かりにくいことがある。」

  → **Gap 発見: Step 2 → 3 のトラブルシューティングガイドがない。よくあるエラーと修正手順を文書化すべき。**

千石: 「Step 4 の `@eval default` がどこまで自動化するかも明確にすべきです。四則演算は自動？ 文字列連結は？ カスタム関数は？」

  → **Gap 発見: @eval default の自動化範囲の仕様書がない。何が自動で何が手動かのマトリクスが必要。**

ヤン: 「Step 6 の VSIX パッケージングが一番ドキュメントが薄い。TextMate grammar の書き方、`package.json` の設定、LSP サーバーのバンドル方法 — 全部散らばってる。」

  → **Gap 発見: VSIX パッケージングの手順書がない。TextMate grammar 生成、package.json テンプレート、ビルドスクリプトのドキュメント。**

リヴァイ: 「各 Step で『成功した状態』を明確にしろ。Step 3 が成功したら何が生成されているはずか。Step 5 が成功したらテストは何件通るはずか。チェックポイントがないと LLM は迷子になる。」

  → **Gap 発見: 各 Step の成功条件（チェックポイント）が未定義。**

## Scene 4: チュートリアルの続きとして書けないか

今泉: 「quickstart-dialogue.ja.md で TinyCalc のチュートリアルがありますよね。その続きとして書けませんか？」

ヤン: 「いいアイデアだね。段階的に複雑にしていく。」

```
Level 1: TinyCalc（既存）— 四則演算だけ
Level 2: TinyCalc + 変数 + if 式 — 条件分岐と変数参照
Level 3: オリジナル DSL — 自分の言語を作る
Level 4: VSIX 化 — エディタ拡張として配布
```

千石: 「Level 2 が最も重要です。変数の導入で『スコープ』の概念が出てくる。if 式で『型の不一致』が出てくる。ここで unlaxer の本当の力が試されます。」

今泉: 「誰が困るの？ Level 3 で『自分のオリジナル DSL を作る』って言っても、何を作ればいいか分からない人は？」

ヤン: 「テンプレートを用意しておくべきだね。『設定ファイル DSL』『ルールエンジン DSL』『クエリ DSL』みたいな典型パターン。」

  → **Gap 発見: Level 3 のテンプレート DSL パターンが未定義。典型的なユースケース別のスターターキットが必要。**
  → **Gap 発見: Level 2 → 3 の難易度ジャンプが大きい。中間の Level 2.5（TinyCalc + 文字列操作）があると良い。**

リヴァイ: 「Level 1 から Level 4 まで、一貫した 1 つの言語を段階的に拡張していくべきだ。Level ごとに別の言語を作ると文脈が切れる。」

千石: 「正しい。TinyCalc → TinyCalc with variables → TinyCalc with types → TinyCalc VSIX。一本道が最も学習効率が高い。」

  → **Gap 発見: 段階的チュートリアルの各 Level 間の差分が未設計。差分が小さすぎると冗長、大きすぎると脱落する。**

## Scene 5: LLM 用のプロンプトテンプレート

ヤン: 「LLM に手順書を読ませるだけじゃなくて、プロンプトテンプレートを用意しておくと効率的だね。」

今泉: 「どういうプロンプトですか？」

ヤン: 「3 つのフェーズに分ける。」

```
Prompt 1: 「この言語仕様から UBNF 文法を書いて」
  Input:  言語仕様（自然言語 or BNF）+ unlaxer リファレンス
  Output: .ubnf ファイル

Prompt 2: 「この文法の evalXxx() を実装して」
  Input:  生成された AST + @eval の説明 + 言語のセマンティクス
  Output: Evaluator.java

Prompt 3: 「VSIX をパッケージングして」
  Input:  動作する Parser + Evaluator + VSIX テンプレート
  Output: package.json, textmate grammar, ビルドスクリプト
```

千石: 「各プロンプトに『期待する出力のフォーマット』を明示すべきです。LLM が自由に書くと、プロジェクトの規約と合わない出力になります。」

  → **Gap 発見: 各プロンプトの出力フォーマット仕様が未定義。ファイル名、パッケージ名、コーディング規約の指定が必要。**

リヴァイ: 「プロンプトにバリデーション手順も含めろ。『生成したら `mvn compile` を実行して、エラーがなければ次へ進め』と明記しないと、LLM は生成して満足する。」

  → **Gap 発見: プロンプトにバリデーション手順（コンパイル、テスト実行）が組み込まれていない。**

今泉: 「そもそも、これらのプロンプトテンプレートはどこに置くんですか？ リポジトリの中？ ドキュメントサイト？」

ヤン: 「リポジトリの `docs/prompts/` に置くのが自然だね。LLM に読ませるなら、そのリポジトリの中にあるのが一番確実。」

千石: 「プロンプトのバージョン管理も必要です。unlaxer のバージョンが上がればプロンプトも更新が必要。対応バージョンを明記すべきです。」

  → **Gap 発見: プロンプトテンプレートのバージョン管理方針が未定義。unlaxer のバージョンとの対応関係。**

## Scene 6: MVP 判断

ヤン: 「MVP は明確だ。『LLM 向け unlaxer リファレンス』1 ファイルと、Prompt 1（UBNF 文法生成）のテンプレート。これだけで LLM は文法を書けるようになる。」

リヴァイ: 「リファレンスの内容を確認しろ。(1) UBNF 構文の最小サンプル、(2) @mapping の Before/After、(3) @leftAssoc の使い方、(4) よくある間違い 5 選、(5) mvn compile のエラー対処法。」

千石: 「加えて、成功条件を明記してください。『このリファレンスを Claude に渡して TinyCalc を再現できたら MVP 成功』です。」

今泉: 「要するに、MVP の検証方法は『LLM に渡してパーサーが作れるか試す』ってことですね。」

ヤン: 「そう。再現実験が検証方法そのもの。リファレンスを渡して、TinyCalc 相当のものを 1 回のプロンプトで作れるかどうか。」

  → **Gap 発見: MVP の検証プロトコル（どの LLM で、どういう条件で、何回試すか）が未定義。**
