# DGE Session: UBNF パスと手書きパスの機能パリティ — 同じ言語を二通りで実現するための確立されたやり方

## テーマ
tinyexpression には同一言語を実現する2つのパスがある: UBNF 生成パスと手書きパス。この二重構造は設計なのか技術的負債なのか。パリティの定義、検証方法、移行戦略、新機能追加ワークフローを明らかにし、共存のコストを持続可能にする。

## キャラクター
- ☕ ヤン・ウェンリー — 「要らなくない？」「最もシンプルな解は？」
- 🎩 千石武 — 「品質基準を示す」「ユーザーのために」
- ⚔ リヴァイ兵長 — 「汚い。動くもの見せろ。」
- 👤 今泉慶太 — 「そもそも」「誰が困るの」

## 前提条件
- tinyexpression は月10億トランザクションの本番環境で稼働中
- 手書きパス: NumberExpressionParser, BooleanExpressionParser 等を手動実装。production tested。
- UBNF パス: tinyexpression-p4.ubnf → GGP で Parser/AST/Mapper/Evaluator を自動生成
- AstEvaluatorCalculator が両パスをオーケストレーション: P4 パス優先、失敗時に手書きへフォールバック
- 5つの実行バックエンドが存在（compile-hand, compile-ast, interpret-hand, interpret-ast, java-code）
- LSP/DAP は P4 パスでのみ動作
- 既存のパリティテスト: ThreeExecutionBackendParityTest, AstEvaluatorBackendParityTest, P4BackendParityTest

## 先輩 (ナレーション)
tinyexpression は歴史的に手書きパーサーで始まった。NumberExpressionParser が数式を、BooleanExpressionParser がブール式を処理し、compile-hand バックエンドが本番で月10億トランザクションを捌いている。その後、LSP/DAP サポートのために UBNF 文法定義（tinyexpression-p4.ubnf）と GGP コード生成パイプラインが追加された。P4 パスは文法定義から Parser, AST sealed record, Mapper, Evaluator を自動生成する。現在、AstEvaluatorCalculator が P4 パスを試し、失敗したら手書きパスにフォールバックするハイブリッド戦略をとっている。問題は、2つのパスの機能範囲が一致していないことだ。手書きにはあるが UBNF にない機能、UBNF にはあるが手書きにない機能がそれぞれ存在する。

## Scene 1: そもそもなぜ二つのパスがあるのか

今泉: 「そもそも、同じ言語のパーサーが2つあるのは変じゃないですか？ 普通、パーサーって1つですよね。なぜ2つ作ったんですか？」

ヤン: 「歴史的経緯だよ。手書きが先で、本番で動いてた。P4 は後から来た。LSP で補完やエラー表示をやるには、手書きパーサーの Token 列じゃなくて、ちゃんとした AST が必要だった。手書きを AST 対応に書き換えるより、文法から生成した方が早かったんだろう。」

千石: 「重要な点があります。手書きパスは月10億トランザクションで検証済みです。P4 パスはまだそのレベルの信頼性を得ていません。本番のトラフィックを支えているのは手書きパスです。」

今泉: 「要するに、手書きは『動く実績』、P4 は『ツール対応』ってことですよね。じゃあ、2つあるのは設計上の判断なんですか？ それとも移行途中なんですか？」

ヤン: 「作者は『今のところ消すつもりはない。UBNF版をそのうち正にする』と言ってる。つまり移行途中だけど、期限がない。」

今泉: 「期限がないってことは、永遠に2つのパスをメンテし続ける可能性があるってことですよね。誰が困るんですか？」

  → **Gap 発見: G-01 二重パスの存在が「設計判断」なのか「移行途中の技術的負債」なのか明文化されていない。判断が曖昧なまま共存している。メンテナーが1人の場合、二重メンテの持続可能性が不明。**

千石: 「困るのはメンテナーです。新機能を追加するたびに2つのパスに入れなければならない。入れ忘れたら、フォールバックのおかげで表面上は動くが、パリティが静かに崩れていく。」

  → **Gap 発見: G-02 フォールバック機構がパリティの崩壊を隠蔽する。P4 パスが失敗しても手書きパスが拾うため、P4 パスの機能不足が顕在化しない。**

## Scene 2: パリティの定義

千石: 「パリティとは何を意味するか、明確にすべきです。2つのパスが『同じ』とは、何がどのレベルで同じであるべきなのか。」

ヤン: 「レベル分けしよう。」

```
レベル 1: パース・パリティ
  同じ入力文字列に対して、両パスが同じように成功/失敗する。
  → 受理する言語の集合が一致。

レベル 2: AST パリティ
  同じ入力に対して、意味的に等価な AST が生成される。
  → 型は異なる（手書き: Token 列, P4: sealed record）が、構造が対応。

レベル 3: 評価パリティ
  同じ入力・同じ変数束縛に対して、同じ評価結果が返る。
  → 数値の精度差 (double vs BigDecimal) は許容するか？

レベル 4: コード生成パリティ
  同じ入力に対して、生成される Java コードが同等に動作する。
  → 文字列一致ではなく、動作の等価性。
```

今泉: 「そもそも、全レベルで一致する必要があるんですか？ レベル 3（評価パリティ）だけで十分ってことはないですか？」

千石: 「ユーザーにとって重要なのはレベル 3 です。同じ式を渡して同じ答えが返る。それが保証されていれば、内部の AST が違っても構いません。」

リヴァイ: 「甘いな。レベル 1 が合わないと、P4 パスで parse できない式が手書きに黙ってフォールバックする。パフォーマンス特性が変わる。エラーメッセージが変わる。LSP が動かない式が出る。レベル 1 が基盤だ。」

  → **Gap 発見: G-03 パリティの各レベル（パース、AST、評価、コード生成）の定義と、各レベルで許容される差異の基準が文書化されていない。**

今泉: 「レベル 2 の『意味的に等価な AST』って、どうやって検証するんですか？ 手書きの Token 列と P4 の sealed record を比較する方法がなければ検証できないですよね。」

  → **Gap 発見: G-04 手書きパスの Token 列と P4 パスの sealed record AST を比較する仕組みがない。NumberGeneratedAstAdapter のような変換層はあるが、全式パターンをカバーしているか不明。**

ヤン: 「全レベルのパリティを一度に追求するのは非現実的だ。まずレベル 3（評価パリティ）を全式パターンで保証して、レベル 1（パース・パリティ）はギャップの棚卸しから始めるのが現実的だろう。」

## Scene 3: 新機能追加時のワークフロー

リヴァイ: 「新しい機能を追加するとき、手書きと UBNF の両方に入れるのか。それとも片方だけか。標準が決まっていないなら決めろ。」

ヤン: 「今のやり方を整理しよう。UBNF に機能を追加するとき:」

```
1. tinyexpression-p4.ubnf に文法ルールを追加
2. @mapping アノテーションで AST マッピングを定義
3. mvn generate-sources で Parser/AST/Mapper を再生成
4. GGP concrete class で Evaluator ロジックを実装
5. テストを書いて P4 パスで動作確認
6. 手書きパスにも追加？ → ???
```

今泉: 「6番が??? になってますよね。手書きにも入れるんですか？ 入れないんですか？ 前もそうだったっけ？」

リヴァイ: 「最近の追加を見ると、UBNF にだけ入れて手書きには入れていない。external invocations, @mapping annotations, @eval — これらは P4 パスにしかない。逆に、hand-written にしかない機能もある。Side effect expressions、String concatenation (+)、isPresent() とか。」

千石: 「つまり、新機能追加の標準ワークフローが存在しないということです。追加するたびに、メンテナーの判断で片方か両方に入れている。これは品質管理の失敗です。」

  → **Gap 発見: G-05 新機能追加時の標準ワークフローが定義されていない。UBNF パスと手書きパスのどちらに追加するか、両方に追加するかの判断基準がない。**

今泉: 「誰が困るんですか？ 具体的に。」

リヴァイ: 「テストを書く人間が困る。P4 パスのテストを書いて通った。でも手書きパスでは動かない。AstEvaluatorCalculator のフォールバックのおかげで表面上は動く。だがバックエンドを切り替えたときに壊れる。」

千石: 「ユーザーも困ります。LSP で補完候補に出てくる関数が、compile-hand バックエンドでは動かない。LSP は P4 パスなので P4 にある機能を補完しますが、実行は手書きパスかもしれない。」

  → **Gap 発見: G-06 LSP の補完候補と実行バックエンドの機能範囲が不一致になる可能性がある。LSP は P4 パスの文法に基づいて補完するが、実行が手書きパスの場合、補完された式が実行時に失敗する。**

## Scene 4: パリティテストの現状

今泉: 「パリティを保証するテストは今どのくらいあるんですか？ そもそも、パリティテストという概念はあるんですか？」

千石: 「存在します。3つのテストクラスがあります。」

```
1. ThreeExecutionBackendParityTest
   → 3つのバックエンド（compile-hand, compile-ast, interpret-hand）の結果を比較

2. AstEvaluatorBackendParityTest
   → AST 評価パスと JavaCode 生成パスの結果を比較

3. P4BackendParityTest
   → P4 パスと手書きパスの結果を比較
```

今泉: 「全式パターンをカバーしているんですか？ 例えば、三項演算子、文字列スライシング、side effect expressions、全部入ってます？」

リヴァイ: 「入っていない。パリティテストに入っている式は基本的な数式とブール式だけだろう。エッジケースは手薄だ。」

  → **Gap 発見: G-07 パリティテストのカバレッジが不明。全式パターン（数値演算、ブール演算、三項演算子、文字列操作、関数呼び出し、変数参照、side effects、external invocations 等）をカバーしているか棚卸しが必要。**

今泉: 「新機能を追加したとき、パリティテストに自動的にケースが追加される仕組みはあるんですか？」

ヤン: 「ないだろうね。手動で追加しない限りパリティテストは増えない。」

  → **Gap 発見: G-08 新機能追加時にパリティテストケースを自動追加する仕組みがない。UBNF に文法ルールを追加しても、対応するパリティテストは手動で書かなければならない。忘れるとパリティの崩壊が検出されない。**

千石: 「問題はもう一つあります。パリティテストが失敗したとき、どちらのパスが『正しい』のですか？ P4 パスの結果を正とするのか、手書きパスの結果を正とするのか。」

今泉: 「そもそも、正しい答えを定義する仕様書はあるんですか？」

  → **Gap 発見: G-09 パリティテスト失敗時のオラクル（どちらが正しいか）が未定義。手書きパスは本番実績があるため事実上のオラクルだが、P4 パスが「正」になる移行シナリオでは逆転する。**

## Scene 5: 移行戦略

ヤン: 「最終的に手書きパスを廃止して P4 だけにするのか、それとも共存し続けるのか。紅茶を飲みながら考えよう。」

今泉: 「作者は『UBNF版をそのうち正にする』って言ってましたよね。『そのうち』っていつですか？」

ヤン: 「判断基準が定義されていない。『いつ P4 が正になるか』の条件が不明だ。」

  → **Gap 発見: G-10 P4 パスが「正」になるための判断基準が未定義。パリティ達成率、テストカバレッジ、パフォーマンスベンチマーク、本番投入期間など、何を満たせば移行できるかが不明。**

千石: 「移行には段階があるべきです。」

```
Stage 0 (現在): 手書きが正。P4 は補助。フォールバックで共存。
Stage 1: P4 が手書きの全機能をカバー。パリティテスト 100%。
Stage 2: 本番で P4 パスを一定期間 shadow mode で稼働。差異を検出。
Stage 3: P4 パスをプライマリに昇格。手書きがフォールバック。
Stage 4: 手書きパスを deprecated にする。
Stage 5: 手書きパスを削除。
```

リヴァイ: 「Stage 1 に到達するのに何ヶ月かかる。手書きにしかない機能のリストを見ろ。Side effect expressions, Time range functions, String predicates のドット形式, String slicing, isPresent(), And/Or/Xor, String concatenation。全部 UBNF に追加してテストを通すのか？」

今泉: 「他にないの？ 全部 UBNF に移すんじゃなくて、手書きにしかない機能のうち本番で使われているものだけ移す、とか。」

ヤン: 「それが現実的だ。全機能の移行じゃなく、本番で使われている式パターンの移行。使われていない機能は移行する必要がない。」

  → **Gap 発見: G-11 手書きパスの全機能のうち、本番で実際に使われている機能の棚卸しがない。移行対象の優先順位付けができない。**

千石: 「もう一つ重要な観点があります。LSP/DAP は P4 パスでしか動きません。compile-hand（本番）は手書きパスです。つまり、開発環境と本番環境で異なるパスを使っているということです。」

  → **Gap 発見: G-12 開発環境（LSP/DAP → P4 パス）と本番環境（compile-hand → 手書きパス）で異なるパスを使っている。開発時に LSP で valid と表示された式が、本番では別のパスで処理される。動作が異なる可能性がある。**

## Scene 6: 手書きにあって UBNF にない機能（ギャップの棚卸し）

千石: 「ギャップの棚卸しをしましょう。まず手書きにあって UBNF にない機能から。」

リヴァイ: 「リストを出す。」

```
手書きにあるが UBNF にない:
  1. Side effect expressions — 副作用付き式の評価
  2. Time range functions — 時間範囲関数
  3. String predicates のドット形式 — $name.startsWith('he') 等
  4. String slicing — 文字列スライス
  5. isPresent() — 変数の存在チェック
  6. And/Or/Xor (& | ^) — ビット演算風の論理演算
  7. String concatenation (+) — 文字列結合

UBNF にあるが手書きにない:
  1. External invocations — 外部関数呼び出し
  2. @mapping annotations — AST マッピング
  3. @eval — 評価アノテーション
```

今泉: 「7つ対3つですか。手書きの方が機能が多いんですね。そもそも、この差分リストは正確なんですか？ 他に漏れていないですか？」

  → **Gap 発見: G-13 手書きパスと UBNF パスの機能差分の完全なリストが存在しない。上記リストは推測ベースであり、コードの網羅的な比較から導出されたものではない。**

千石: 「特に懸念なのは String concatenation (+) です。`'hello' + ' ' + 'world'` は非常に基本的な操作です。これが UBNF パスで動かないとしたら、P4 パスの LSP で文字列結合を含む式を書いたユーザーは混乱します。」

ヤン: 「And/Or/Xor も気になるね。`&` `|` `^` は手書きパスでは論理演算子として実装されているが、UBNF ではどうなってる？ `&&` `||` は UBNF にあるだろうが、`&` `|` `^` の短縮形は？」

  → **Gap 発見: G-14 論理演算子の表記揺れ。手書きパスは `&` `|` `^` をサポート、UBNF パスは `&&` `||` をサポート。片方にしかない表記がある場合、パース・パリティが崩れる。**

今泉: 「isPresent() って何ですか？ 変数が定義されているかチェックする関数？ これは UBNF の文法で表現できるんですか？」

リヴァイ: 「isPresent() は変数参照のメタ情報だ。パーサーの領域ではなく、Evaluator の領域。文法には `isPresent` `(` VariableRef `)` を追加して、Evaluator で変数バインディングの有無をチェックすればいい。」

  → **Gap 発見: G-15 isPresent() のような Evaluator 依存の関数を UBNF で表現する方法。文法ルールとしては追加できるが、Evaluator の変数バインディング機構との連携が必要。手書きパスでの実装方法と P4 パスでの実装方法が異なる可能性。**

## Scene 7: 共存をサポートする仕組み

今泉: 「2パスの共存を楽にする仕組みは今どんなものがあるんですか？」

千石: 「3つの仕組みがあります。」

```
1. @TinyAstNode アノテーション
   手書きパーサーに AST マッピングのヒントを付与する。
   P4 パスの AST 構造と対応付けるためのブリッジ。

2. AstEvaluatorCalculator の fallback chain
   P4 パスを試行 → 失敗 → 手書きパスにフォールバック。
   ユーザーからは透過的。

3. NumberGeneratedAstAdapter
   手書きパスの Token 列 AST を P4 パスの sealed record AST に変換する。
   両パスの AST を統一して Evaluator を共有するためのアダプター。
```

今泉: 「要するに、フォールバック、アノテーション、アダプターの3つですね。これで十分なんですか？ 他に何が要りますか？」

ヤン: 「十分かどうかは、目的による。共存を続けるなら、もう1つ必要だ。『P4 パスでパースできるが手書きではパースできない式』と『手書きでパースできるが P4 ではパースできない式』を検出する仕組み。」

  → **Gap 発見: G-16 パース・パリティの差異を自動検出する仕組みがない。ある式が P4 では通るが手書きでは通らない（またはその逆）ことを CI で検出できない。**

リヴァイ: 「NumberGeneratedAstAdapter は数式だけだろう。ブール式、三項演算子、文字列式のアダプターはあるのか？」

  → **Gap 発見: G-17 AST 変換アダプター（NumberGeneratedAstAdapter）が全式カテゴリをカバーしているか不明。数式以外（ブール式、三項演算子、文字列式）のアダプターが存在しない可能性。**

千石: 「fallback chain にも問題があります。フォールバックが発生したことをログに記録していますか？ Silent fallback は問題の発見を遅らせます。」

  → **Gap 発見: G-18 AstEvaluatorCalculator の fallback 発生時のログ/メトリクスがあるか不明。Silent fallback はパリティ崩壊の検出を遅らせる。本番でどの程度 fallback が発生しているか可視化されていない。**

## Scene 8: UBNF-first ワークフローの提案

リヴァイ: 「新しい機能は UBNF に入れるのが先。手書きは後。これを標準にしろ。」

ヤン: 「根拠は？」

リヴァイ: 「UBNF に入れれば、Parser、AST、Mapper が自動生成される。手書きで全部書くより圧倒的に速い。GGP concrete class で Evaluator ロジックだけ書けばいい。手書きパスは本番で必要な場合にだけ追加する。」

千石: 「ワークフローを明文化しましょう。」

```
UBNF-first ワークフロー:
  1. tinyexpression-p4.ubnf に文法ルールを追加
  2. @mapping で AST マッピングを定義
  3. mvn generate-sources で再生成
  4. GGP concrete class で Evaluator ロジック実装
  5. P4 パスのテストを書いて green 確認
  6. パリティテストを追加: P4 の結果を正として記録
  7. 手書きパスが必要な場合のみ追加
  8. 手書きを追加した場合、パリティテストで P4 と同じ結果になることを検証
```

今泉: 「でも、今は逆ですよね。手書きが正で、P4 が追従してる。ワークフローを逆転させるタイミングはいつですか？」

  → **Gap 発見: G-19 UBNF-first ワークフローへの切り替えタイミングと条件が未定義。現在のオラクル（手書きが正）を P4 に切り替えるには、全既存機能の P4 パス実装完了が前提条件。**

ヤン: 「切り替えは段階的にやればいい。新機能は UBNF-first。既存機能は手書きが正のまま。2つのルールが同時に存在する期間がある。」

リヴァイ: 「2つのルールが同時に存在すると混乱する。新機能は P4 が正、既存機能は手書きが正。テストの assertion の基準が機能ごとに異なる。」

  → **Gap 発見: G-20 移行期間中のオラクル管理。機能ごとに「どちらのパスが正か」が異なる過渡期の管理方法が未定義。テストフレームワークでこれを表現する仕組みが必要。**

## Scene 9: ツール/自動化の提案

ヤン: 「手動で二重メンテするのはダメだ。自動化できるものは自動化しろ。」

リヴァイ: 「3つの自動化を提案する。」

```
自動化 1: パリティテスト自動生成
  UBNF の全ルールに対して、P4 vs 手書きのテストケースを自動生成する。
  文法ルールから式パターンを列挙し、両パスに投入して結果を比較。

自動化 2: カバレッジレポート
  「この式パターンは P4 で通るが手書きで通らない」の自動検出。
  「この式パターンは手書きで通るが P4 で通らない」の自動検出。
  CI で毎回実行し、差分を可視化。

自動化 3: 手書きパーサーのスケルトン生成
  UBNF 文法から手書きパーサーのスケルトンコード（メソッドシグネチャ + TODO コメント）を生成。
  GGP の新しいジェネレーターとして実装する可能性。
```

今泉: 「自動化 1 は、文法ルールから式パターンを列挙するんですよね。それって、property-based testing に近くないですか？ ランダムな式を生成して両パスに通す。」

千石: 「property-based testing は有効です。ただし、式の生成が文法的に valid であることを保証する必要があります。無効な式を生成しても意味がない。」

  → **Gap 発見: G-21 UBNF 文法からの valid 式自動生成器が存在しない。パリティテストの自動生成には、文法ルールに基づいた valid 式の網羅的生成が必要。これは GGP の機能として実現可能か。**

ヤン: 「自動化 2 のカバレッジレポートが最も投資対効果が高いと思う。今すぐ作れるし、差分が見えるだけで優先順位付けが楽になる。」

リヴァイ: 「自動化 3 は後回しだ。手書きパーサーのスケルトン生成は、手書きパスを今後も維持する前提の投資だ。手書きを廃止する方針なら要らない。」

  → **Gap 発見: G-22 自動化への投資判断が移行戦略に依存する。手書きパスを維持するなら自動化 3 に価値があるが、廃止するなら不要。移行戦略が未決定のまま自動化に投資すると、使われないツールを作る可能性。**

## Observe → Suggest → Act

### O-S-A 1: 機能差分の棚卸し

- **Observe**: 手書きパスと P4 パスの機能差分リストが存在しない。どの式パターンがどちらのパスで動くか把握できていない。
- **Suggest**: 両パスのパーサー/Evaluator のコードを網羅的に比較し、機能差分の完全リストを作成する。
- **Act**: 手書きパスのパーサー（NumberExpressionParser, BooleanExpressionParser 等）の全対応演算子・関数と、UBNF 文法の全ルールを突合して差分表を作成。各差分について「本番使用あり/なし」を可能な範囲で注記。

### O-S-A 2: パリティテストの拡充

- **Observe**: 既存パリティテストのカバレッジが不明。基本的な式しかテストされていない可能性。
- **Suggest**: 全式カテゴリ（数値演算、ブール演算、三項演算子、文字列操作、関数呼び出し、変数参照、side effects）のパリティテストケースを追加する。
- **Act**: 既存の ThreeExecutionBackendParityTest, P4BackendParityTest のテストケースを棚卸し。カバーされていない式カテゴリを特定し、テストケースを追加。各テストケースに「この機能は P4/手書き/両方 で対応済み」のアノテーションを付与。

### O-S-A 3: Fallback ログの追加

- **Observe**: AstEvaluatorCalculator の fallback 発生時のログが不明。Silent fallback はパリティ崩壊の検出を遅らせる。
- **Suggest**: Fallback 発生時に、式の内容、失敗理由、フォールバック先をログに記録する。
- **Act**: AstEvaluatorCalculator の fallback 箇所に logging を追加。ログレベルは WARN（本番で見えるように）。ログ内容: 入力式（先頭 N 文字）、P4 パスの失敗理由（例外クラス + メッセージ）、フォールバック先のバックエンド名。

### O-S-A 4: パース・パリティの自動検出

- **Observe**: ある式が P4 で通るが手書きで通らない（またはその逆）ことを自動検出する仕組みがない。
- **Suggest**: CI に「パース・パリティチェック」ジョブを追加。既存テストの式パターンを両パスに投入し、パース成功/失敗の差異をレポートする。
- **Act**: 既存テストケースの全式を収集。各式を P4 パスと手書きパスの両方に投入。結果が異なる式のリストを出力。このリストを CI artifact として保存し、差分の増減を追跡する。

### O-S-A 5: UBNF-first ワークフローの文書化

- **Observe**: 新機能追加の標準ワークフローが存在しない。メンテナーの判断で片方か両方に追加されている。
- **Suggest**: UBNF-first ワークフローを CONTRIBUTING.md またはプロジェクトドキュメントに明文化する。
- **Act**: 以下を文書化: (1) 新機能は UBNF に追加 → GGP 再生成 → concrete class 実装 → P4 テスト。(2) パリティテストを追加（P4 の結果を正として記録）。(3) 手書きパスへの追加は、本番の compile-hand バックエンドで必要な場合のみ。(4) 手書きを追加した場合、パリティテストで検証。
