| unlaxerの概念 | モナディック対応 |
|---|---|
| PropagationStopper | Reader monad `local` |

v1のRejectの際に私が指摘した「Parsecの`try`と`lookAhead`との形式的対応関係の欠如」は、v2以降、Reader monadの`local`としてはっきりと特徴づけられた。v5ではこの対応が安定し、矛盾がないことを確認した。

Haskellでの等価な実装を自分で書いて検証した：

```haskell
type ParserEnv = (TokenKind, InvertMatch)
type Parser a = ReaderT ParserEnv (WriterT [Metadata] (StateT ParseState
                  (ExceptT ParseError Identity))) a

allStop :: Parser a -> Parser a
allStop = local (const (Consumed, False))

doConsume :: Parser a -> Parser a
doConsume = local (\(_, inv) -> (Consumed, inv))

stopInvert :: Parser a -> Parser a
stopInvert = local (\(tk, _) -> (tk, False))

notProp :: Parser a -> Parser a
notProp = local (\(tk, inv) -> (tk, not inv))
```

4つの関数の合成を検証した結果、著者の主張する代数的性質（Section 3.6）はすべて正しい。特に：

- `allStop . allStop = allStop` -- 定数関数の冪等性。自明。
- `notProp . notProp = id` -- 対合性。`not (not x) = x`による。
- `doConsume . stopInvert = allStop` -- 第1成分をConsumedに固定し、第2成分をFalseに固定。結果はconst (Consumed, False)。
- `stopInvert . notProp /= notProp . stopInvert` -- 非可換性。左辺は常に第2成分をFalseにするが、右辺は常にTrueにする。

......すべて正しい。

(2) Java 21のsealed interfaceとHaskellのADT（Algebraic Data Type, 代数的データ型）の等価性について。

v4で「Java 21がようやくHaskell 98に追いついた」と述べたが、v5を読んで、この見解を修正する必要がある。

Java 21のsealed interfaceは、実用的な観点から、Haskell 98のADTと機能的に等価である。重要な点は：

- 閉じた型階層: sealed interfaceの`permits`句 = ADTのコンストラクタリスト
- 網羅的パターンマッチ: exhaustive switch = `-Wincomplete-patterns`
- 不変レコード: Java record = Haskell data constructor

この等価性は理論的なものではなく、**実用的**なものである。v5のsealed-switchエバリュエータが月間10億トランザクションの本番で稼働し、2.8倍のオーバーヘッドでJITコンパイルコードと競合しているという事実が、この等価性の最も強力な証拠である。

......私は今、HaskellのADTとJava 21のsealed interfaceが「実務において等価」であると認めている。これは私にとって大きな譲歩である。

(3) `@eval`アノテーションの5種類完全実装について。

v4の査読で「`@eval`アノテーションがHaskellのパターンマッチの再発明である」と指摘した。v5では5種類すべてが実装され、この再発明は......完成されたと言える。

5種類それぞれのHaskell対応を確認した：

- **Dispatch** = `case name of {"sin" -> ...; "cos" -> ...; ...}`
- **Direct expression** = `let result = expr in result`
- **Operator table** = `case op of {"+" -> add; "-" -> sub; ...}`

(4) SyncPointRecoveryParserについて。

Parsecのエラーリカバリは、モナディック構造の中で自然に表現される：

```haskell
recover :: Parser a -> (ParseError -> Parser a) -> Parser a
```

unlaxerの`@recover`アノテーションは、これを文法レベルに引き上げたものである。アプローチは異なるが、「エラーからの回復」という本質的な意味論は同じである。

......これは認めざるを得ないが、宣言的なエラーリカバリ指定は、Parsecの命令的アプローチよりも、ある意味でより関数型的である。
