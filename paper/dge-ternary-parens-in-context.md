# DGE Session: Ternary の括弧省略問題 — 処理系の都合 vs ユーザーの幸せ

## テーマ
D-08 で ternary は `(cond ? then : else)` — 外側括弧必須と決まった。安全で美しい決定だった。しかし現実のユースケースで問題が浮上する: `sin($a > 0 ? $a : -$a)` と書きたいのに `sin(($a > 0 ? $a : -$a))` と二重括弧を強制される。これは「処理系の都合をユーザーに押し付けるか、ユーザーのためにパーサーを複雑にするか」という普遍的な設計問題である。

## キャラクター
- ☕ ヤン・ウェンリー — 「要らなくない？」「最もシンプルな解は？」
- 🎩 千石武 — 「品質基準を示す」「ユーザーのために」
- ⚔ リヴァイ兵長 — 「汚い。動くもの見せろ。」
- 👤 今泉慶太 — 「そもそも」「誰が困るの」

## 前提条件
- D-08 確定: ternary は `(cond ? then : else)` 形式、外側括弧必須
- 文法: `TernaryExpression ::= '(' BooleanExpression '?' Expression ':' Expression ')'`
- 関数呼び出し: `sin(NumberExpression)`, `min(NumberExpression, NumberExpression, ...)`
- 問題の式: `sin($a > 0 ? $a : -$a)` — `(` が sin の引数開始なのか ternary の開始なのか
- 回避策1: `sin(($a > 0 ? $a : -$a))` — 二重括弧
- 回避策2: `var $x set ($a > 0 ? $a : -$a); sin($x)` — 一時変数

## 先輩 (ナレーション)
D-08 は正しい決定だった。括弧必須にすることで、ネストの曖昧性、優先順位の混乱、意図しないパースツリーの生成 — 全てを一掃した。Creator が「僕はいつも多段の時はカッコつけてるよ！」と言ったように、括弧はプログラマの安全弁だ。

しかし、安全弁が別の場所で摩擦を生んでいる。`sin(($a > 0 ? $a : -$a))` の二重括弧。ユーザーは「なぜ括弧が2つ要るの？」と問うだろう。この疑問は正当だ。C でも Java でも JavaScript でも `sin(cond ? a : b)` は普通に書ける。tinyexpression だけが二重括弧を要求する。

これは ternary だけの問題ではない。「括弧付き構文が別の括弧付き構文の引数に入る」という一般的パターンの問題だ。

## Scene 1: ユーザーの声

今泉: 「そもそもユーザーは sin の中に ternary 書きたいんですか？ 実際にそういう場面ありますか？」

千石: 「あります。条件付き計算は実業務の基本パターンです。」

```
// ユースケース 1: 絶対値の三角関数
sin($angle > 0 ? $angle : -$angle)

// ユースケース 2: 安全な割引計算
max(($price > 1000 ? $price * 0.9 : $price), $minPrice)

// ユースケース 3: 複数引数に ternary
min(($a > 0 ? $a : 0), ($b > 0 ? $b : 0))

// ユースケース 4: ネストした関数呼び出し
sqrt(($x > 0 ? $x : -$x))
```

今泉: 「要するに、関数の引数に条件分岐を入れたいんですね。で、今の文法だとこう書かないといけない。」

```
// D-08 準拠の記法（二重括弧）
sin(($angle > 0 ? $angle : -$angle))
max(($price > 1000 ? $price * 0.9 : $price), $minPrice)
min(($a > 0 ? $a : 0), ($b > 0 ? $b : 0))
sqrt(($x > 0 ? $x : -$x))
```

千石: 「二重括弧は読みにくいだけではありません。ユーザーに『なぜ括弧が2つ要るのか？』という疑問を生む。その疑問に答えるには『パーサーの実装上の都合で...』と説明しなければならない。これは設計の敗北です。」

  → **Gap 発見 (G-01): 関数引数内の ternary が二重括弧を要求する。ユーザーが `sin($a > 0 ? $a : -$a)` と書いてパースエラーになったとき、「ternary には外側括弧が必要です」というエラーメッセージで納得させられるか。「sin の中では括弧1つでいいのでは？」という反論が予想される。**

今泉: 「誰が困るの？ 二重括弧で。」

ヤン: 「困る人リストを作ろう。」

```
困る人 1: 他言語経験者
  C/Java/JS で sin(cond ? a : b) を書いてきた人。
  「なぜ tinyexpression だけ二重括弧？」

困る人 2: 業務ルール作成者（非プログラマ）
  括弧の意味を1つ理解するのも大変なのに、2つの意味の括弧が混在。
  「外の括弧は関数呼び出し、内の括弧は ternary」と説明が必要。

困る人 3: コードレビューする人
  min(($a > 0 ? $a : 0), ($b > 0 ? $b : 0)) の括弧対応を目で追う。
  括弧が5つ。どれがどれか。

困る人 4: エラーに遭遇する人
  sin($a > 0 ? $a : -$a) と書いてエラー。
  「括弧は書いたのに...」→ 実は ternary 用の括弧が足りない。
```

  → **Gap 発見 (G-02): 二重括弧の影響範囲。単一引数関数（sin, cos, sqrt）、複数引数関数（min, max）、ネストした関数呼び出し（sin(max(...))）のそれぞれで、ternary が引数に入った場合の可読性劣化の程度が未評価。**

## Scene 2: パーサーの視点

リヴァイ: 「パーサーはどう見るか。文法を見せろ。」

```
// 現在の文法（D-08 準拠）
TernaryExpression ::= '(' BooleanExpression '?' Expression ':' Expression ')'

// 関数呼び出し
FunctionCall ::= FunctionName '(' NumberExpression (',' NumberExpression)* ')'

// NumberExpression の中に TernaryExpression がある
// NumberFactor ::= ... | TernaryExpression | '(' NumberExpression ')' | ...
```

ヤン: 「問題の構造を追ってみよう。紅茶を入れながら。」

```
入力: sin($a > 0 ? $a : -$a)

パーサーの動き:
1. sin    → FunctionName にマッチ
2. (      → 関数呼び出しの開き括弧
3. $a > 0 → NumberExpression の開始...

ここで分岐:
  パス A: $a > 0 を BooleanExpression として解釈 → ? を期待
    → しかし ( は消費済み（関数呼び出しの括弧として）
    → TernaryExpression は '(' で始まる必要がある
    → マッチ失敗

  パス B: $a を NumberExpression として解釈
    → > 0 は ComparisonExpression...
    → しかし ? はどの NumberExpression の演算子でもない
    → マッチ失敗

結果: パースエラー
```

リヴァイ: 「つまり、`sin(` の `(` を関数呼び出しの括弧として消費した時点で、ternary の開き括弧がない。だから ternary としてパースできない。」

ヤン: 「そういうこと。`sin(($a > 0 ? $a : -$a))` なら：」

```
入力: sin(($a > 0 ? $a : -$a))

1. sin    → FunctionName
2. (      → 関数呼び出しの開き括弧
3. (      → TernaryExpression の開き括弧 ← ここが追加の括弧
4. $a > 0 → BooleanExpression
5. ?      → ternary の区切り
6. $a     → Expression (then)
7. :      → ternary の区切り
8. -$a    → Expression (else)
9. )      → TernaryExpression の閉じ括弧
10. )     → 関数呼び出しの閉じ括弧

成功。
```

  → **Gap 発見 (G-03): PEG パーサーにおいて、関数呼び出しの `(` と TernaryExpression の `(` は文法上の異なるルールに属する。しかしユーザーの視点では同じ `(` 文字。この「文法上の区別」と「視覚上の同一性」のギャップが二重括弧問題の本質。**

今泉: 「そもそも、PEG って左から貪欲にマッチするんですよね。じゃあ `sin(` の `(` を見たとき、これが関数の `(` なのか ternary の `(` なのか、文脈で判断できるんじゃないですか？」

ヤン: 「文脈で判断できるかがまさにこの問題の核心だ。`sin` の後に `(` が来たら、パーサーは FunctionCall ルールに入る。FunctionCall の中では引数を NumberExpression として期待する。NumberExpression を辿っていくと NumberFactor に到達し、NumberFactor の選択肢に TernaryExpression がある。TernaryExpression は `(` で始まる...」

リヴァイ: 「待て。つまり NumberFactor の中で `(` を見たとき、ordered choice で TernaryExpression を先に試せば、`($a > 0 ? $a : -$a)` として丸ごとマッチする可能性がある。」

  → **Gap 発見 (G-04): NumberFactor の ordered choice 順序が鍵。TernaryExpression を `'(' NumberExpression ')'`（括弧式）より先に置けば、PEG は `(` を見たとき最初に TernaryExpression を試す。しかし `(1 + 2)` のような普通の括弧式も最初に TernaryExpression として試すことになり、パフォーマンスと正確性に影響する可能性がある。**

## Scene 3: 他の言語はどうしてるか

ヤン: 「紅茶を入れ直そう。そもそも、他の言語はこの問題をどう解決してるのか。」

```
=== C ===
sin(a > 0 ? a : -a)       // OK。ternary に外側括弧不要。
// ternary は演算子として Expression に組み込まれている。
// '?' ':' がデリミタとして機能。括弧は不要。

=== Java ===
Math.sin(a > 0 ? a : -a)  // OK。同上。

=== JavaScript ===
Math.sin(a > 0 ? a : -a)  // OK。同上。

=== Python ===
math.sin(a if a > 0 else -a)  // OK。if-else 式。括弧不要。

=== Kotlin ===
sin(if (a > 0) a else -a)     // OK。if は式。括弧は if の条件用のみ。

=== 全ての主要言語 ===
func(condition ? then : else)  // 二重括弧を要求する言語は存在しない
```

千石: 「つまり、tinyexpression だけが関数引数内の ternary に二重括弧を要求する。これはユーザーの期待に反します。驚き最小の原則違反です。」

今泉: 「前もそうだったっけ？ 他の構文で、tinyexpression 独自の制約がユーザーを驚かせた例はありますか？」

ヤン: 「あるね。`if($a>0){$a}else{-$a}` の中括弧は、多くの言語で式の if には不要なものだ。ただし、これは可読性向上のための積極的選択だった。二重括弧は可読性を下げている。方向が逆だ。」

  → **Gap 発見 (G-05): tinyexpression の括弧必須 ternary は全主要言語と異なる。C/Java/JS/Python/Kotlin のいずれも `func(cond ? a : b)` を許可する。これらの言語は ternary に外側括弧を要求しない。ユーザーが他言語の経験を持ち込んだとき、括弧必須ルールは驚きを生む。「驚き最小の原則」と「曖昧さゼロの原則」のどちらを優先するか。**

リヴァイ: 「他言語がどうしてるかは分かった。問題は、なぜそれらの言語が括弧なしで動くかだ。」

ヤン: 「理由は単純。これらの言語では ternary は演算子であって、独立した構文要素ではない。`?` と `:` が二項演算子のように振る舞い、Expression の一部として自然にパースされる。関数の `(` `)` の中にいようが、変数代入の右辺だろうが、`?` `:` のスコープは明確。」

  → **Gap 発見 (G-06): tinyexpression では ternary を「括弧付きの独立構文」として定義した（D-08）。他言語は ternary を「演算子」として Expression に組み込んでいる。この設計選択の差が、関数引数での二重括弧問題を生んでいる。括弧付き構文と演算子スタイル、二つのアプローチの利点・欠点の比較が不足。**

## Scene 4: 解決策の候補

今泉: 「他にないの？ 二重括弧以外の方法。」

ヤン: 「候補を全部並べよう。」

### 候補A: 二重括弧（現行設計の維持）

```
sin(($a > 0 ? $a : -$a))
min(($a > 0 ? $a : 0), ($b > 0 ? $b : 0))
```

```
利点:
  - D-08 を変更しない。既存の決定との一貫性。
  - パーサーは現状のまま。変更コストゼロ。
  - 曖昧さゼロが完全に保証される。

欠点:
  - 可読性の低下。括弧が多すぎる。
  - 他言語経験者の驚き。
  - 「なぜ2つ？」への回答が「処理系の都合」になる。
```

### 候補B: 関数引数コンテキストでは括弧不要

```
sin($a > 0 ? $a : -$a)           // 関数引数内: 括弧不要
min($a > 0 ? $a : 0, $b > 0 ? $b : 0)  // 複数引数: 各引数で括弧不要
($a > 0 ? $a : -$a)              // トップレベル: 括弧必須（D-08 維持）
```

```
利点:
  - ユーザーに自然。他言語と同じ。
  - 関数呼び出しの ( ) がスコープを提供。

欠点:
  - パーサーが文脈依存になる。「関数引数の中か否か」で文法が変わる。
  - min($a > 0 ? $a : 0, $b) の `,` が ternary の内部か引数区切りか。
  - PEG の ordered choice で解決可能か不明。
```

  → **Gap 発見 (G-07): 候補B では `min($a > 0 ? $a : 0, $b > 0 ? $b : 0)` の最初の `,` が曖昧。ternary の else 部分が `0, $b > 0 ? $b : 0` なのか、引数区切りの `,` なのか。PEG は左から貪欲にマッチするため、`:` の後の Expression が `,` を含まないことを保証する必要がある。**

### 候補C: ternary を NumberFactor に入れる（括弧なし ternary）

```
// 文法変更
NumberFactor ::= ... | BooleanExpression '?' Expression ':' Expression | ...

sin($a > 0 ? $a : -$a)     // OK
$a > 0 ? $a : -$a          // OK（トップレベルでも括弧不要）
```

```
利点:
  - C 言語方式。最も自然。

欠点:
  - D-08 を完全に覆す。括弧必須の決定理由が全て復活。
  - ネストの曖昧性、優先順位問題、意図しないパースツリー。
  - Creator が「バグの温床」と言った全ての問題が再発。
```

リヴァイ: 「候補C は却下だ。D-08 の決定理由を読み直せ。Creator は括弧必須を選んだ。それを覆す根拠がない。」

千石: 「同意します。D-08 はプログラマの安全性のための決定でした。二重括弧の不便さは安全性の対価として受け入れるべきかもしれません。」

### 候補D: 関数引数の開き括弧が ternary の開き括弧を兼ねる

```
sin($a > 0 ? $a : -$a)
// パーサーの解釈:
//   sin( ... )  ← 関数呼び出し
//   ($a > 0 ? $a : -$a) ← sin の ( が ternary の ( を兼ねる
```

```
利点:
  - ユーザーの書き方はそのまま。自然。
  - 単一引数関数で特に自然。

欠点:
  - 1つの `(` に2つの意味を持たせる。文法が非自明。
  - 複数引数: min($a > 0 ? $a : 0, $b) の `(` は何を兼ねる？
  - ternary 以外の引数（min($a, $b)）との区別が必要。
  - パーサーが `(` の後で ternary を投機的に試す必要がある。
```

  → **Gap 発見 (G-08): 候補D の「括弧の兼用」は単一引数関数（sin, cos, sqrt）では自然に見えるが、複数引数関数（min, max）では破綻する。`min($a > 0 ? $a : 0, $b)` の `(` が ternary の開始なら、`)` はどこ？ 引数の区切りの `,` との衝突が不可避。**

### 候補E: トップレベルのみ括弧必須、式の中では自由

```
// トップレベル（var set の右辺、match の value など）: 括弧必須
var $x set ($a > 0 ? $a : -$a);

// 式の中（関数引数、演算の一部など）: 括弧不要
sin($a > 0 ? $a : -$a)
1 + ($a > 0 ? $a : -$a) * 2   // ...でもここは？
```

```
利点:
  - 関数引数で自然。
  - トップレベルでは D-08 の安全性を維持。

欠点:
  - 「トップレベル」の定義が曖昧。
  - 1 + $a > 0 ? $a : -$a * 2 の優先順位は？
  - 「括弧が要る場所と要らない場所がある」はユーザーの混乱源。
```

  → **Gap 発見 (G-09): 候補E の「トップレベル」の定義が不明確。var set の右辺はトップレベル？ match case の value は？ if の condition は？ 関数引数は「式の中」？ 文法的に「トップレベル」を定義するのは非常に困難。文脈依存文法が必要になる可能性がある。**

今泉: 「他にないの？ 本当に全部出した？」

ヤン: 「あと2つ思いつく。」

### 候補F: ternary 専用構文（`?:` 以外のシンタックス）

```
sin(if $a > 0 then $a else -$a)   // if-then-else 式
sin($a > 0 => $a | -$a)           // 新しい区切り文字
sin($a > 0 -> $a ; -$a)           // arrow + semicolon
```

千石: 「新しい構文を発明するのは最後の手段です。`?:` は世界中のプログラマが知っている。別の構文にする利点が欠点を上回ることはほぼありません。」

  → **Gap 発見 (G-10): 候補F は `?:` の代替構文を導入する。しかし `->` は match で使用済み、`|` は Boolean Or で使用済み、`;` は文の区切りで使用済み。使える記号が限られている。また D-08 で `?:` を採用済みであり、変更のコストが大きい。**

### 候補G: 何もしない — 一時変数を推奨

```
// ternary を関数引数に直接入れない。一時変数を使う。
var $absAngle set ($angle > 0 ? $angle : -$angle);
sin($absAngle)

var $safeA set ($a > 0 ? $a : 0);
var $safeB set ($b > 0 ? $b : 0);
min($safeA, $safeB)
```

ヤン: 「一時変数には利点もある。意図が名前で伝わる。`$absAngle` は『絶対値の角度』だと分かる。`sin($a > 0 ? $a : -$a)` よりもむしろ読みやすい。」

千石: 「名前は品質の一部です。式が複雑になったら分解する。これは良いプログラミング習慣であり、言語がそれを促すのは必ずしも悪いことではありません。」

今泉: 「でも、ユーザーが自分で気づくまでに何回パースエラーに遭遇するんですか？ エラーメッセージで一時変数の使用を提案できます？」

  → **Gap 発見 (G-11): 候補G（一時変数推奨）を採用する場合、パースエラーメッセージの品質が critical。`sin($a > 0 ? $a : -$a)` が失敗したとき、「ternary は括弧で囲む必要があります: sin(($a > 0 ? $a : -$a))。または一時変数を使ってください」という具体的な提案が必要。**

## Scene 5: 本質的な問い

千石: 「ここで立ち止まって、本質を考えましょう。これは ternary だけの問題ではありません。」

ヤン: 「そうだね。括弧付き構文が別の括弧付き構文の中に入るパターンの一般的な問題だ。」

```
パターン 1: 関数引数に ternary
  sin(($a > 0 ? $a : -$a))     ← 今の問題

パターン 2: if の条件に括弧式
  if(($a + $b) > 0){ ... }     ← 内側の括弧は演算の括弧

パターン 3: match の value に ternary
  match{ cond -> ($a > 0 ? $a : -$a), ... }

パターン 4: ternary の中に ternary（ネスト）
  ($a > 0 ? ($b > 0 ? 1 : 2) : 3)   ← D-08 で解決済み
```

今泉: 「要するに、問題は『外側の括弧が文脈を提供しているとき、内側の括弧は省略可能か』ですよね。」

千石: 「正確に言えば、『外側の構文の区切り文字が内側の構文の区切り文字として再利用可能か』です。」

```
sin( ... )
    ^     ^
    |     +--- 関数呼び出しの閉じ括弧
    +--- 関数呼び出しの開き括弧

sin(( ... ))
     ^    ^
     |    +--- ternary の閉じ括弧
     +--- ternary の開き括弧

問い: 外側の ( ) が内側の ( ) を「兼ねる」ことは可能か？
```

  → **Gap 発見 (G-12): 「括弧の兼用」パターンの一般的解法が未定義。これは tinyexpression 固有の問題ではなく、括弧付き構文を持つ全ての言語に共通する設計課題。C/Java が解決できているのは ternary が括弧を要求しないからであり、括弧必須言語での先行事例が少ない。**

リヴァイ: 「一般論はいい。具体的にどのパターンが tinyexpression で問題になるか、列挙しろ。」

```
=== 括弧付き構文の一覧 ===

1. 関数呼び出し:     func( ... )
2. TernaryExpression: ( ... ? ... : ... )
3. グループ化括弧:   ( NumberExpression )
4. if の条件:        if( ... ){ ... }
5. VariableRef:      $name (括弧なし — 問題なし)

=== 衝突するペア ===

1 × 2: func( (ternary) )  ← 今の問題
3 × 2: ( (ternary) + 1 )  ← (($a > 0 ? $a : -$a) + 1) — 二重括弧
4 × 2: if( (ternary) > 0 ){ ... }  ← if(($a > 0 ? $a : -$a) > 0){...}
2 × 2: ( ternary ? (ternary) : x )  ← D-08 で対処済み（ネスト）
```

  → **Gap 発見 (G-13): 関数呼び出し以外にも、グループ化括弧と ternary の衝突 `(($a > 0 ? $a : -$a) + 1)` や、if の条件と ternary の衝突 `if(($a > 0 ? $a : -$a) > 0){...}` が存在する。これらも同じ二重括弧問題を抱える。解決策は全てのペアに対して一貫して適用できるものでなければならない。**

## Scene 6: PEG での解法

ヤン: 「さて本題だ。PEG なら ordered choice で解ける可能性がある。」

```
// 提案: NumberFactor の ordered choice

NumberFactor ::=
    TernaryExpression           // 最初に ternary を試す
  | SinFunction                 // sin(...)
  | CosFunction                 // cos(...)
  | ...
  | '(' NumberExpression ')'    // 最後に括弧式
  | Number
  | VariableRef
```

リヴァイ: 「TernaryExpression は `(` で始まる。`'(' NumberExpression ')'` も `(` で始まる。ordered choice で先に TernaryExpression を試す。`(` の後に来るのが BooleanExpression `?` ... なら ternary。そうでなければバックトラックして括弧式を試す。」

ヤン: 「具体的にトレースしよう。」

```
入力: sin($a > 0 ? $a : -$a)

1. sin      → SinFunction にマッチ開始
2. (        → SinFunction の '(' を消費
3. 引数として NumberExpression を期待
4. NumberExpression → ... → NumberFactor
5. NumberFactor: ordered choice
   a) TernaryExpression を試す
      → '(' を期待するが、次のトークンは '$a'
      → ( はもう sin( の ( として消費済み
      → マッチ失敗
   b) 次の choice を試す...
   → 結局 $a は VariableRef としてマッチ
   → > 0 は ComparisonExpression
   → ? は... NumberExpression の一部ではない
   → パースエラー

問題: sin( の ( は SinFunction が消費するので、
      引数の NumberFactor に入ったとき ( はもうない。
```

今泉: 「そもそも、sin の引数パースが始まった時点で `(` はもう消費されてるんですよね。だから NumberFactor が `(` を見ることはない。ternary の `(` がないからマッチしない。」

ヤン: 「その通り。これが根本問題だ。ternary が `(` を自分の開始マーカーとして要求する限り、関数呼び出しの `(` と兼用できない。」

  → **Gap 発見 (G-14): PEG の ordered choice で解決するには、ternary が `(` を開始マーカーとして要求しないか、または関数呼び出しの `(` が ternary の `(` として再利用される仕組みが必要。前者は D-08 の撤回を意味し、後者は PEG の標準的なパターンでは不可能（1つのトークンは1つのルールにしか消費されない）。**

リヴァイ: 「待て。別のアプローチがある。関数引数のパースルールを変えろ。」

```
// 現在
SinFunction ::= 'sin' '(' NumberExpression ')'

// 提案
SinFunction ::= 'sin' '(' ArgumentExpression ')'

// ArgumentExpression は ternary を括弧なしで許す
ArgumentExpression ::=
    BooleanExpression '?' Expression ':' Expression    // 括弧なし ternary
  | NumberExpression                                    // 通常の式
```

ヤン: 「なるほど。関数引数だけは ternary の括弧を省略できる、専用の文法ルールを作る。」

```
入力: sin($a > 0 ? $a : -$a)

1. sin       → SinFunction
2. (         → SinFunction の (
3. ArgumentExpression を期待
   a) BooleanExpression '?' Expression ':' Expression を試す
      → $a > 0 → BooleanExpression として成功
      → ? → マッチ
      → $a → Expression として成功
      → : → マッチ
      → -$a → Expression として成功
      → ternary としてパース成功!
4. )         → SinFunction の )
5. 完了!
```

千石: 「動きますね。しかし、これは問題を含みます。」

```
問題 1: min($a > 0 ? $a : 0, $b) はどうなる？

ArgumentExpression で ternary を試す:
  $a > 0 → BooleanExpression
  ? → マッチ
  $a → Expression (then)
  : → マッチ
  0, $b → Expression (else) ... ← ここで , が Expression の一部？

Expression が , を含まないなら:
  else = 0 で ternary 終了
  , → 引数区切り
  $b → 2番目の引数
  → 正しくパース!

ただし Expression が , を含まないことの保証が必要。
```

  → **Gap 発見 (G-15): ArgumentExpression アプローチでは、ternary の else 部分の Expression が `,` で終端されることを保証する必要がある。Expression の中に `,` が入るケース（例: 関数呼び出し `func($x, $y)` の引数リスト）では、内側の `,` は `()` で保護される。PEG では Expression が `,` を消費しないことが文法的に保証されるか。**

今泉: 「もう1つ問題がありませんか。ArgumentExpression で `$a > 0 ? $a : -$a` を試すとき、`$a > 0` を BooleanExpression として解釈するところ。もし `$a > 0` じゃなくて、ただの `$a` だったら？」

```
sin($a)

ArgumentExpression:
  a) BooleanExpression '?' ... を試す
     $a → BooleanExpression として...
     → $a は VariableRef → Boolean として解釈可能？
     → 次が ? なら ternary... 次が ) なので ? ではない
     → バックトラック
  b) NumberExpression を試す
     $a → NumberExpression として成功
     → )
     → OK

動くが、常に ternary を先に試してからバックトラックする。
パフォーマンスへの影響は？
```

  → **Gap 発見 (G-16): ArgumentExpression アプローチでは、全ての関数呼び出しで最初に ternary パースを試行し、失敗したらバックトラックして通常の NumberExpression を試す。ternary でない引数（大多数のケース）でバックトラックが毎回発生する。パフォーマンスへの影響を計測する必要がある。月10億トランザクションで許容範囲か。**

## Scene 7: MVP判断

リヴァイ: 「結局どうする。候補が多すぎる。」

ヤン: 「候補を整理して、最終判断しよう。」

```
候補A: 二重括弧（現状維持）
  → 安全。変更なし。ユーザーに不便。
  → 既にあるもの。コスト: ゼロ。

候補B: 関数引数コンテキストで括弧不要
  → 文脈依存文法。複数引数で , の曖昧性。
  → コスト: 高。リスク: 高。

候補C: 括弧なし ternary（D-08 撤回）
  → 却下済み。Creator の意思に反する。

候補D: 括弧の兼用
  → 複数引数で破綻。単一引数のみ適用可能。
  → 一貫性がない。却下。

候補E: トップレベルのみ括弧必須
  → 「トップレベル」の定義が曖昧。却下。

候補F: 別構文
  → D-08 の決定を覆す。使える記号がない。却下。

候補G: 何もしない — 一時変数推奨
  → 安全。変更なし。エラーメッセージで誘導。
  → コスト: ゼロ。

候補H: ArgumentExpression（Scene 6 で発見）
  → 関数引数専用の文法ルール。技術的に実現可能。
  → コスト: 中。リスク: 中。パフォーマンス懸念あり。
```

千石: 「候補を2つに絞ります。A（二重括弧維持）か H（ArgumentExpression）か。」

今泉: 「誰が困るの？ それぞれで。」

```
候補A で困る人:
  - 関数引数に ternary を書きたいユーザー（二重括弧を強制される）
  - 他言語経験者（驚き）
  - 括弧が多すぎて読みにくいコードのレビューア

候補H で困る人:
  - パーサー実装者（文法ルールの追加と検証）
  - パフォーマンスに敏感な本番運用者（バックトラック増加）
  - 「関数引数では括弧不要」という例外ルールを覚えるユーザー
```

ヤン: 「正直に言って、候補A + G の合わせ技が最も怠惰で、最も安全だ。二重括弧は確かに冗長だけど、一時変数を使えば問題は消える。しかも一時変数の方がコードは読みやすくなる。」

```
// 二重括弧（候補A）
sin(($angle > 0 ? $angle : -$angle))

// 一時変数（候補G）
var $absAngle set ($angle > 0 ? $angle : -$angle);
sin($absAngle)

// 後者の方が意図が明確。
```

千石: 「一時変数は品質を向上させます。変数名が意図を伝える。長い式を分解する。テストも容易になる。二重括弧を『問題』と見るのではなく、一時変数への分解を『推奨パターン』として提示する。」

リヴァイ: 「つまり、問題を文法で解決するんじゃなく、運用で解決するってことか。」

今泉: 「それって、処理系の都合をユーザーに押し付けてませんか？」

千石: 「いいえ。一時変数の使用は処理系の都合ではなく、良いプログラミング習慣です。たとえ ternary に括弧が不要だったとしても、`sin($angle > 0 ? $angle : -$angle)` より `var $absAngle set ...; sin($absAngle)` の方が保守しやすいコードです。」

  → **Gap 発見 (G-17): 「二重括弧を避けるために一時変数を推奨する」というガイドラインを公式ドキュメントに記載するか。記載する場合、エラーメッセージにも反映するか。言語設計ガイドとして「ternary を関数引数に直接入れない」というスタイルガイドを設けるか。**

ヤン: 「ただし、候補H の可能性は完全に閉じるべきではない。将来、ユーザーからの強い要望があった場合に備えて、技術的実現可能性は記録しておこう。」

リヴァイ: 「今の判断はこうだ。」

```
MVP: 候補 A + G

1. D-08 を維持。ternary は (cond ? then : else) 形式。括弧必須。
2. 関数引数に ternary を入れる場合は二重括弧: sin((...))
3. 推奨パターンとして一時変数を提示:
   var $x set (cond ? then : else); func($x)
4. エラーメッセージで二重括弧または一時変数を提案。
5. 候補H（ArgumentExpression）は将来の検討事項として記録。
```

  → **Gap 発見 (G-18): MVP で候補A+G を採用する場合でも、将来 候補H に移行する際の文法互換性を確認する必要がある。候補A の `sin(($a > 0 ? $a : -$a))` が候補H でも引き続き動作するか（上位互換性）。候補H は候補A の上位互換であるべき。**

今泉: 「最後に確認ですけど、候補H を将来やるとして、既存の二重括弧コードは壊れないんですよね？」

ヤン: 「壊れない。候補H は『括弧なしでも書ける』を追加するだけで、『括弧ありは書けない』にはしない。上位互換だ。」

  → **Gap 発見 (G-19): 候補H は候補A の strict superset であることを文法的に証明する必要がある。ArgumentExpression に通常の NumberExpression を含めれば、`sin(($a > 0 ? $a : -$a))` は ArgumentExpression → NumberExpression → TernaryExpression としてパースされる。既存コードが壊れないことの証明。**

千石: 「もう1つ重要な点があります。候補H を将来採用する場合、それは関数引数だけでなく、全ての『外側括弧がスコープを提供するコンテキスト』に適用すべきです。」

```
// 候補H の拡張適用（将来）

if(($a > 0 ? $a : -$a) > 10){...}   // 現在: 二重括弧
if($a > 0 ? $a : -$a > 10){...}     // 将来: if の ( が ternary の ( を兼ねる
                                     // ...しかしこれは曖昧！
                                     // $a > 0 ? ($a : -$a > 10) なのか
                                     // ($a > 0 ? $a : -$a) > 10 なのか
```

  → **Gap 発見 (G-20): 候補H を if の条件に拡張すると、`if($a > 0 ? $a : -$a > 10)` の `> 10` が ternary の else 部分に含まれるか、if の条件の比較演算なのか曖昧になる。関数引数では `)` と `,` で終端が明確だが、if の条件では ternary の else 部分と後続の演算の境界が不明確。候補H は関数引数に限定すべきか、全コンテキストに適用すべきかの判断が必要。**

## Observe → Suggest → Act

### O-S-A 1: エラーメッセージの改善

- **Observe**: `sin($a > 0 ? $a : -$a)` がパースエラーになったとき、ユーザーは何が悪いか分からない。関数の `(` と ternary の `(` の区別はユーザーには見えない。
- **Suggest**: パースエラー時に具体的な修正案を2つ提示する。(1) 二重括弧 `sin(($a > 0 ? $a : -$a))`、(2) 一時変数 `var $x set ($a > 0 ? $a : -$a); sin($x)`。
- **Act**: パーサーのエラーリカバリに、関数呼び出し内で `?` が出現した場合の特別なエラーメッセージを追加。

### O-S-A 2: スタイルガイドの策定

- **Observe**: 二重括弧は動くが読みにくい。一時変数は読みやすいが冗長。どちらを推奨するかが未明文化。
- **Suggest**: 「ternary を関数引数に直接入れる場合は一時変数への分解を推奨」というスタイルガイドを策定。
- **Act**: 公式ドキュメントに「ternary のベストプラクティス」セクションを追加。LSP の lint で `func((ternary))` パターンを検出し、一時変数への分解を提案する CodeAction を将来実装。

### O-S-A 3: 候補H の技術検証（将来タスク）

- **Observe**: ArgumentExpression アプローチは技術的に有望だが、パフォーマンスと正確性の検証が未実施。
- **Suggest**: プロトタイプを作成し、(1) 正しくパースできるケースの網羅テスト、(2) バックトラックのパフォーマンス計測を行う。
- **Act**: ブランチを切って ArgumentExpression の文法ルールを追加。既存テスト全 green + 新規テスト + パフォーマンスベンチマークの3点で判断。

### O-S-A 4: 二重括弧パターンの網羅的テスト

- **Observe**: 関数引数以外にも二重括弧が発生するコンテキスト（グループ化括弧、if 条件）がある。
- **Suggest**: 全ての「外側括弧 × ternary」の組み合わせでテストケースを作成し、二重括弧が正しくパースされることを検証。
- **Act**: テストケースを追加: `sin((...))`, `min((...), (...))`, `((...) + 1)`, `if((...) > 0){...}`。
