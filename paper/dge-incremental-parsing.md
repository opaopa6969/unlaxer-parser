# DGE Session: インクリメンタル構文解析の設計

## テーマ
tinyexpression の数十KB規模の式に対するインクリメンタル構文解析の設計ギャップを発見する

## キャラクター
- ☕ ヤン・ウェンリー — 「要らなくない？」「最もシンプルな解は？」
- 🎩 千石武 — 「品質基準を示す」「ユーザーのために」
- ⚔ リヴァイ兵長 — 「汚い。動くもの見せろ。」
- 👤 今泉慶太 — 「そもそも」「誰が困るの」

## 背景
- tinyexpression の式は数十KBに達する（数百文字ではない）
- 典型例: match{ 47都道府県 × 10カテゴリ = 470 case } → 約23KB
- 本番環境: 月間10億トランザクション（UDF として実行）
- PEG ベースのパーサー（バックトラックあり）
- 現状: 毎回フルリパース
- LSP レスポンス目標: < 100ms
- 50KB の式 → フルパース約100ms → LSP としてボーダーライン

---

## Scene 1: The Real Problem（数十KBの衝撃）

今泉: 「そもそもどのくらい大きいんですか？ 実データは？ 前の DGE で『数百文字』って話だったと思うんですけど。」

ヤン: 「...あれは僕の見積もりが甘かった。紅茶飲みながら適当に言った。実際の本番データを見たら、match 式で47都道府県 × 10カテゴリ = 470 case とかある。テキストにして約23KB。数百文字どころか数万文字だ。」

  → **Gap 発見: 前回 DGE での「数百文字」という見積もりが2桁間違い。実運用データに基づく設計が必要。**

今泉: 「要するに、23KB の式を毎回フルパースしてるってことですよね？ それで100ms。」

ヤン: 「そう。50KB だと100ms くらい。LSP の応答要件が100ms 以内だから、ちょうどボーダーライン。今は動いてるけど、式がもう少し大きくなったら破綻する。」

千石: 「ボーダーラインで『動いている』というのは、品質基準を満たしているとは言えません。余裕を持った設計をすべきです。」

リヴァイ: 「で、実運用での式サイズの分布はどうなんだ。最大23KB なのか、平均23KB なのか。」

  → **Gap 発見: 実運用での式サイズの分布データが不明。平均・中央値・P95・最大値を計測すべき。**

今泉: 「月10億トランザクションって言ってましたけど、UDF として実行するときもパースしてるんですか？」

ヤン: 「いい質問だね。UDF 実行時はパース済みの AST をキャッシュしてるはず。パースが重いのは LSP 側の問題。つまりエディタで式を編集するたびにフルリパースが走る。」

  → **Gap 発見: UDF 実行時の AST キャッシュ戦略と LSP 側のキャッシュ戦略が別物。両者の関係を整理すべき。**

## Scene 2: チャンクベース vs フルインクリメンタル

ヤン: 「tree-sitter みたいなフルインクリメンタルは重い。パーサーの内部構造を全部書き換えることになる。チャンクベースで90%の問題は解決できると思う。」

千石: 「チャンクの境界設計を間違えるとバグの温床です。パースの正しさが保証できなくなります。」

リヴァイ: 「最小限の変更で最大の効果を出せ。tree-sitter のクローンを作るな。」

今泉: 「他にないの？ チャンクベースとフルインクリメンタルの間に何かありませんか？」

ヤン: 「3つの選択肢がある。(1) チャンクベース — カンマや行でテキストを分割して、変更されたチャンクだけ再パース。(2) Packrat キャッシュの部分無効化 — メモ化テーブルの一部だけ捨てて再パース。(3) tree-sitter 型 — 構文木のノードに位置情報を持たせてツリーの差分更新。」

千石: 「(1) はシンプルですが、チャンク境界をまたぐ構文をどう扱いますか？」

ヤン: 「match 式のカンマ区切りなら、各 case は独立したパース単位になる。`$prefecture == "東京", result1` のような1行が1チャンク。」

リヴァイ: 「行単位か、カンマ単位か、AST ノード単位か。どれだ。」

ヤン: 「カンマ区切りが一番自然。match 式の case 境界がカンマだから。var 宣言ならセミコロン。」

  → **Gap 発見: PEG のバックトラックがチャンク境界を超える場合の挙動が未定義。例: ordered choice の最初の選択肢がカンマを超えて消費し、失敗してバックトラックするケース。**

今泉: 「そもそも、カンマで区切れない式はあるんですか？」

ヤン: 「ネストした match 式。`match(x, case1, match(y, case2a, case2b), case3)` みたいに、case の値として別の match が来る。内側の match のカンマと外側のカンマを区別する必要がある。」

  → **Gap 発見: ネストした match 式でのチャンク境界の決定方法。括弧の深さを追跡する必要がある。**

千石: 「括弧の深さを追跡するだけでは不十分です。文字列リテラル内のカンマは無視しなければなりません。`"Hello, World"` の中のカンマでチャンクを分割してはいけません。」

  → **Gap 発見: 文字列リテラル内のカンマを無視するためのトークンレベルの前処理が必要。チャンク分割は raw text ではなくトークン列に対して行うべきか？**

## Scene 3: キャッシュ戦略

今泉: 「キャッシュのキーは何ですか？」

ヤン: 「一番シンプルなのは hash(case text) → parsed Token。case のテキストが同じなら、パース結果も同じ。」

千石: 「テキストが同じでもパース結果が異なる場合はありませんか？」

ヤン: 「...ある。変数宣言が case の前にあると、変数のスコープが case のパースに影響する可能性がある。`var x = 1; match(...)` で、case 内の `$x` の解釈が変わる。」

  → **Gap 発見: 変数宣言が変わったら全 case の再パースが必要か？ 変数スコープがパース結果に影響する範囲の特定が必要。**

今泉: 「要するに、キャッシュキーにはスコープ情報も含めないといけないってことですよね？」

ヤン: 「正確に言うと、tinyexpression のパースフェーズでは変数の解決はしない。パースは構文だけ。変数の解決は評価フェーズ。だからパースのキャッシュキーは純粋にテキストだけでいいかもしれない。」

千石: 「本当にそうですか？ パーサーが変数名の存在をチェックしていませんか？」

  → **Gap 発見: tinyexpression のパーサーが変数名の存在チェックをパースフェーズで行っているか未確認。パースとセマンティクスの境界を明確にすべき。**

リヴァイ: 「キャッシュの invalidation はどうする。LRU か、時間ベースか。」

ヤン: 「LSP のユースケースなら、開いてるファイルのキャッシュだけ持てばいい。ファイルを閉じたらキャッシュも捨てる。LRU は不要。」

  → **Gap 発見: キャッシュの invalidation 戦略が未定義。ファイル単位か、ワークスペース単位か、エディタセッション単位か。**

今泉: 「前もそうだったっけ？ キャッシュが古くて間違った結果が出たことは？」

ヤン: 「UDF のキャッシュでは経験がある。式を更新したのにキャッシュが残ってて旧バージョンの結果が出た。invalidation のバグは一番見つけにくい。」

  → **Gap 発見: キャッシュの正当性検証メカニズムが必要。キャッシュヒット時にもフルパースの結果と一致することをデバッグモードで検証できるべき。**

## Scene 4: Token の immutability 問題

千石: 「Token が mutable なのは設計として問題です。filteredChildren を後から追加できるということは、パース結果が後から変わるということです。キャッシュと相性が最悪です。」

ヤン: 「まあ、そうなんだけど、今さら immutable にしたら全部壊れるよ。Token を使ってる箇所は... 数え切れない。」

リヴァイ: 「壊れてもいいから正しく直せ。mutable な状態を共有するのは最悪の設計だ。」

今泉: 「そもそも、なんで mutable にしたんですか？」

ヤン: 「パース後にフィルタリングをかけて、不要なトークン（空白やコメント）を除外する処理がある。`filteredChildren` はそのために後から設定される。パース結果を加工する後処理フェーズがあるんだ。」

  → **Gap 発見: Token の immutable 化の影響範囲が不明。filteredChildren を使っている全箇所の洗い出しが必要。**

千石: 「後処理フェーズの結果もキャッシュに含めるべきです。パース結果 + フィルタリング結果をセットでキャッシュすれば、Token が mutable でも問題ありません。」

ヤン: 「それだと後処理の変更があったときにキャッシュが無効になる。でも後処理のロジックはめったに変わらないから、実用上は問題ないか。」

リヴァイ: 「2つの選択肢だ。(A) Token を immutable にする。正しい道。工数大。(B) Token は mutable のまま、キャッシュには後処理済みの結果を入れる。妥協。工数小。」

  → **Gap 発見: immutable にしなくてもインクリメンタル解析は実装可能か？ mutable Token + defensive copy で安全性を保てるか？**

今泉: 「誰が困るの？ Token が mutable で実際に困った事例は？」

ヤン: 「...今のところバグとしては顕在化してない。でも並行パースをやろうとしたら確実に壊れる。LSP は非同期だから、将来的には並行パースの需要がある。」

  → **Gap 発見: LSP の非同期処理で Token の mutable 性が問題になるタイミング。現時点では問題ないが、並行パースを導入する際に破綻する。**

## Scene 5: Packrat Cache の活用

ヤン: 「Packrat parsing のメモ化テーブルをそのまま使えないかな。Packrat cache は (position, parser) → result のマッピング。テキストの一部が変わったら、変更位置以降のキャッシュだけ invalidate すればいい。」

千石: 「理論的にはそうですが、position はテキスト内のオフセットですよね？ テキストの途中に1文字挿入したら、その位置以降の全エントリの position がずれます。」

  → **Gap 発見: Packrat cache の position はテキストオフセットベース。挿入/削除で全 position がシフトするため、単純な部分 invalidation では不十分。position の再マッピングが必要。**

ヤン: 「position を行番号 + 列番号にすれば、変更行以外は影響を受けない... いや、PEG のバックトラックがあるからそう単純じゃないか。」

今泉: 「そもそも、unlaxer の ParseContext に Packrat cache はあるんですか？」

ヤン: 「...確認しないと分からない。unlaxer は PEG ベースだから Packrat の仕組みは入れられるはずだけど、今の実装にメモ化テーブルがあるかどうかは別の話だ。」

  → **Gap 発見: unlaxer の ParseContext に Packrat cache（メモ化テーブル）が既に存在するか未確認。存在しない場合、まず Packrat 化が先行タスクになる。**

リヴァイ: 「確認してこい。話はそれからだ。」

千石: 「Packrat cache の導入自体にもコストがあります。メモリ使用量が O(入力長 × パーサー数) になる。50KB の式に対して、パーサーが100個あれば 5MB のメモ化テーブルです。」

  → **Gap 発見: Packrat cache のメモリオーバーヘッド。50KB × パーサー数 のメモリが許容できるか？ LSP プロセスのメモリ制約は？**

今泉: 「他にないの？ Packrat 以外でメモ化する方法は？」

ヤン: 「AST ノードレベルでのキャッシュがある。パースされた Token ツリーのサブツリーを再利用する。変更されたノードの祖先だけ再構築して、兄弟ノードはそのまま使う。tree-sitter がまさにこの方式。」

  → **Gap 発見: AST ノードレベルキャッシュと Packrat キャッシュの比較評価が必要。どちらが unlaxer のアーキテクチャに適合するか。**

## Scene 6: LSP との統合

千石: 「LSP の textDocument/didChange イベントでは差分情報が来ます。`contentChanges: [{range, text}]` という形式で、変更範囲が正確に分かります。」

今泉: 「要するに、どこが変わったかは LSP が教えてくれるんですね？ じゃあ自分で diff を取る必要はない？」

千石: 「そうです。ただし2つのモードがあります。incremental と full。incremental なら差分だけ来ますが、full なら全文が来ます。サーバー側で incremental を宣言する必要があります。」

  → **Gap 発見: LSP サーバーが textDocumentSync を incremental モードで宣言しているか？ full モードなら差分情報が得られず、自前で diff を取る必要がある。**

ヤン: 「contentChanges から変更範囲が分かったとして、その範囲がどのチャンクに対応するかを特定する必要がある。カンマ区切りの case 470個のうち、変更が case 235 に含まれるなら、case 235 だけ再パースすればいい。」

今泉: 「その『対応するチャンクの特定』はどうやるんですか？」

ヤン: 「前回のパース結果から、各 case の開始位置と終了位置を覚えておく。変更範囲の start position が case 235 の範囲内なら、case 235 が対象。」

  → **Gap 発見: 変更範囲からチャンク（case）を特定するためのインデックス構造が必要。各 case の (start, end) 位置を保持する仕組み。**

千石: 「変更が case の境界そのもの（カンマ）に当たった場合はどうしますか？ カンマを削除したら2つの case がマージされます。」

  → **Gap 発見: チャンク境界（カンマ）自体の編集への対応。case の追加・削除・マージが起きた場合、チャンク構造の再構築が必要。**

リヴァイ: 「パース結果の部分更新をどう Token tree に反映するんだ。case 235 を再パースしたら、新しい Token サブツリーが出来る。それを元の Token tree の case 235 の位置に差し替えるのか。」

ヤン: 「そう。Token tree の match ノードの children[235] を新しいサブツリーで置き換える。ただし Token が mutable だから、置き換えは直接参照を書き換えれば済む... のが良いのか悪いのか。」

  → **Gap 発見: パース結果の部分更新を Token tree に反映する具体的なアルゴリズムが未定義。既存の children リストの要素差し替えで十分か、ツリー全体の再構築が必要か。**

千石: 「Token tree を部分的に更新した場合、position 情報（行番号・列番号・オフセット）の再計算はどうしますか？ case 235 以降の全 Token の position がずれます。」

  → **Gap 発見: 部分更新後の position 再計算コスト。case 235 以降の全 Token の position 更新が O(N) になり、結局フルリパースと同じコストになりうる。遅延評価が必要か？**

## Scene 7: MVP 設計

リヴァイ: 「全部やるな。最小限で動くものを出せ。」

今泉: 「一番効果が大きいのはどこですか？」

ヤン: 「match 式の case 単位キャッシュだけ。470 case のうち1つを変更したら、その1つだけ再パースする。残り469はキャッシュから。これだけで match 式のパース時間が 1/470 になる。」

千石: 「他の DSL 構造は？ var 宣言や if 式は？」

ヤン: 「それらはフルリパースで十分。match 式が23KB で全体の大半を占めてるから、match だけ最適化すれば90%は解決する。他は後で対応すればいい。」

リヴァイ: 「具体的にどう実装する。」

ヤン: 「3ステップ。(1) match 式のパース時に各 case のテキストと Token を記録する。(2) テキスト変更時に変更された case を特定する。(3) その case だけ再パースして Token tree を差し替える。」

今泉: 「それ、どのくらいで作れるんですか？」

ヤン: 「match のパーサーに case キャッシュを追加するだけなら... パーサー側の変更は小さい。問題は LSP 側のインテグレーション。変更通知を受け取って、キャッシュを更新して、パース結果を返す部分。」

  → **Gap 発見: MVP の実装工数の見積もりが不明。パーサー変更 + LSP 統合 + テスト、それぞれの工数。**

千石: 「MVP でも、正しさの保証は必要です。キャッシュを使ったパース結果がフルパースの結果と一致することをテストで保証すべきです。」

  → **Gap 発見: MVP のテスト戦略。インクリメンタルパースの結果とフルパースの結果の一致をどう検証するか。property-based testing が有効か。**

リヴァイ: 「テストのコストも入れて見積もれ。テストなしでリリースするな。」

今泉: 「前もそうだったっけ？ キャッシュを入れて想定外のバグが出た経験は？」

ヤン: 「...さっきも言ったけど、UDF のキャッシュで invalidation バグがあった。キャッシュを入れるなら、キャッシュなしモード（フルリパースにフォールバック）は必ず残すべき。切り替えフラグが必要だ。」

  → **Gap 発見: インクリメンタルモードとフルリパースモードの切り替えフラグが必要。デバッグ時やバグ発生時にフォールバックできるように。**

今泉: 「要するに、MVP は『match 式の case キャッシュ + フォールバック付き』ということですか？」

ヤン: 「そうだね。最小限はそれだ。」

千石: 「もう1つ。MVP の成功指標を定義してください。『100ms が何ms になるか』を測定できなければ、効果があったかどうか分かりません。」

  → **Gap 発見: MVP の成功指標（ベンチマーク）が未定義。フルパース時間 vs インクリメンタルパース時間の計測基盤が必要。**

リヴァイ: 「計測しろ。数字で示せ。話はそれからだ。」
